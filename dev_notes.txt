Logical Flow of Game
---------------------
    Board setup:

        1. We initialize an empty board, represented by an empty dict

        2. We initialize Position objects for that board. Those position objects only have one field - a position
        value represented by algebraic notation (a1, c2, etc). We then populate the dict with Position objects -
        the key being the Position's notation (a str value) and the Value being the Position object itself.

        3. We initialize two Player objects - one with _player_name = "BLUE" and one "RED"

        4. Each player object generates its own 16 pieces.
                Should the Player be tied to the Piece or Piece tied to the player? Or both?

                Two mandatory methods are:

                    make_move("old_pos", "new_pos")

                        1. First step is to confirm if "old_pos" (1) exists (RET false if not) and (2) has
                        the correct player on it (RET false if not).

                            Here, it makes the most sense for a Piece object to be "stored inside" a Position object.


                            That Piece has to be "assigned" to a Player, but only for the purpose of making it so
                            only that player can control it. Thus, a Piece should could have a simple field declaring
                            what player it is assigned to. But, that Piece could also be tied to the actual Player
                            object that its associated with, which might provide more functionality. But I'm unsure
                            if this would increase coupling and/or decrease modularity.

                        2. Second step would be to determine if the new_pos (1) exists (RET false if not) and (2)
                        if that position does not have a piece belonging to the current player (RET false if not).

                        The third step would be to determine if the move is valid - here we would diverge
                        in our decision tree based on if the move is "Reposition" or an "Attempted_Capture". Unsure
                        if these should be separate Move objects (or if we're even going to be using a Move object at
                        all).
