Logical Flow of Game
---------------------
    Board setup:

        1. We initialize an empty board, represented by an empty dict

        2. We initialize Position objects for that board. Those position objects only have one field - a position
        value represented by algebraic notation (a1, c2, etc). We then populate the dict with Position objects -
        the key being the Position's notation (a str value) and the Value being the Position object itself.

        3. We initialize two Player objects - one with _player_name = "BLUE" and one "RED"

        4. Each player object generates its own 16 pieces.
                Should the Player be tied to the Piece or Piece tied to the player? Or both?

                Two mandatory methods are:

                    make_move("old_pos", "new_pos")

                        1. First step is to confirm if "old_pos" (1) exists (RET false if not) and (2) has
                        the correct player on it (RET false if not).

                            Here, it makes the most sense for a Piece object to be "stored inside" a Position object.


                            That Piece has to be "assigned" to a Player, but only for the purpose of making it so
                            only that player can control it. Thus, a Piece should could have a simple field declaring
                            what player it is assigned to. But, that Piece could also be tied to the actual Player
                            object that its associated with, which might provide more functionality. But I'm unsure
                            if this would increase coupling and/or decrease modularity.

                        2. Second step would be to determine if the new_pos (1) exists (RET false if not) and (2)
                        if that position does not have a piece belonging to the current player (RET false if not).

                        3. The third step would be to determine if the move is valid - here our decision tree would
                        diverge based on if the move is "Reposition" or an "Attempted_Capture". Unsure if these should
                        be separate Move objects (or if we're even going to be using a Move object at all).


                    is_in_check("player_color")

                        1. First step is to filter bad input

                        2. Second step is to determine if the player is in check

                            The primary method I can think of for determining if a player is in check is to just
                            have a boolean field in a player object that we could check each time. But this would
                            still involve a method like "check_if_player_is_in_check()" to be run after each player
                            takes their move.

                            So with that said, is there a more efficient way to implement determining a check
                            situation? One thing to remember is that a player can't put themselves in check - so
                            we would need to use a check_for_check() method after we solidify a move that the
                            current player makes. So, if we were to have a move class, check_for_check() might
                            make the most sense here. After all, the concept of "check" isn't a state that can
                            only be associated with a player being in check - a Move is either a move that creates
                            a check situation or doesn't create a check situation.

                            So this lends itself to actually implementing a Move class. I like this because it
                            feels like it will make the class design more modular, and it will actually give us
                            more space to expand the game so that we can track moves over the course of the game
                            or even provide functionality to reverse a move.
